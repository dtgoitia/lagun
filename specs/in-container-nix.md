## Summary

Enable Claude Code, running inside its agent container, to use nix.

## Context

- Claude Code runs inside a container built from an Ubuntu-based Dockerfile generated by `devShells.createShell` in the lagun flake.
- Nix provides the development tooling for human developers on the host via `nix develop`. The agent should have access to the same tooling via the consuming project's `flake.nix`.
- The agent container routes all HTTP/HTTPS traffic through OneCLI (an in-flight secret-swapping proxy). Nix downloads from public endpoints (cache.nixos.org, GitHub) that require no secrets.

## Requirements

- Single-user Nix is installed and available in every agent container (baked into the Dockerfile generated by `devShells.createShell`, not opt-in).
- Nix is configured with `experimental-features = nix-command flakes`.
- No extra binary caches beyond the default (cache.nixos.org). No pinned channels.
- `/nix` is persisted across container runs via a named Podman volume (for speed). The volume name must follow the same per-consumer naming convention as other lagun volumes (e.g. `${name}-lagun-nix`).
- Documentation is provided on how to clear the Nix store volume when needed (e.g. to reclaim disk space or force a clean rebuild).
- `NO_PROXY` is set in the rendered Compose file to exclude Nix-related domains (cache.nixos.org, github.com, \*.github.com) from OneCLI. This is set alongside the existing `HTTP_PROXY`/`HTTPS_PROXY` vars and applies to all tools in the container, not just Nix. The documentation must explain why this traffic bypasses OneCLI: OneCLI performs HTTPS inspection (MITM) to swap secrets in-flight, which would cause Nix TLS verification to fail; and these are public endpoints that carry no secrets, so bypassing OneCLI is both necessary and safe.

## Constraints

- Must work in a rootless Podman OCI container with no systemd â€” single-user Nix (no daemon) is the only viable mode.
- The container runs as root. Nix is installed with `--yes` (suppresses the interactive root check). The `nixbld` group and 32 build users are created manually before running the installer, and `/nix` is created with `mkdir -m 0755` â€” this pre-satisfies the installer's expectations so it proceeds non-interactively. Keeping root avoids forcing consumers to wrap `extraDockerfileLines` apt-get installs in `USER root` / `USER lagun` switches. The security benefit of non-root is modest given the container is already isolated via rootless Podman.
- The `/nix` volume (not `/nix/store`) must be mounted, because Nix uses both `/nix/store` (packages) and `/nix/var` (profiles, gc-roots, state). Mounting only `/nix/store` would leave `/nix/var` ephemeral, breaking `nix develop` across restarts.
- The `NO_PROXY` exclusion is OS-wide (affects curl, wget, etc. too), not scoped to Nix. This is acceptable because the excluded domains are public and carry no secrets.
- Private flake inputs or private binary caches that need secret injection via OneCLI are not supported under this design â€” they would silently bypass OneCLI.
- The agent does not need pre-commit hooks to run. Entering `nix develop` triggers the consuming project's `shellHook`, which typically installs hooks into `.git/hooks/` and writes `.pre-commit-config.yaml` via `git-hooks.nix`. Because `/workspace` is a bind-mounted host directory, those writes propagate to the host. This is prevented by gating `gitHooks.shellHook` on a `LAGUN_AGENT=1` environment variable set in the compose file: the shellHook checks `if [ -z "${LAGUN_AGENT:-}" ]` and skips hook installation when running inside the container.
- The first `nix develop` run after a fresh volume will take several minutes (downloading nixpkgs, building the dev shell). This is expected and acceptable; subsequent runs reuse the cached store.

## Out of scope

- Private binary caches.
- Private flake inputs (e.g. private GitHub repos referenced in `flake.nix`).
- Pinning nixpkgs channels or locking the Nix version to match the host.
- Nix daemon mode or multi-user Nix.

## Implementation

### `flake.nix` â€” `agentInPodman` attrset

Add two new variables alongside `bunVersion`:

```nix
nixVersion = "2.28.3";  # find latest at https://releases.nixos.org/nix/
nixVolumeName = "${name}-lagun-nix";
```

### `flake.nix` â€” `dockerfile` (`base` stage)

1. Add `xz-utils` to the existing `apt-get install` list (required by the Nix installer).

2. After the apt block, add a Nix install block (before the bun block â€” both are pinned, so cache order is stable; Nix before bun is more foundational):

```dockerfile
# install single-user nix
ENV PATH="/root/.nix-profile/bin:/nix/var/nix/profiles/default/bin:${PATH}"
RUN mkdir -p /root/.config/nix \
  && echo "experimental-features = nix-command flakes" > /root/.config/nix/nix.conf \
  && groupadd -g 30000 nixbld \
  && for i in $(seq 1 32); do \
      useradd -u $((30000 + i)) -G nixbld -d /var/empty -s /sbin/nologin nixbld$i; \
    done \
  && mkdir -m 0755 /nix \
  && curl -fsSL https://releases.nixos.org/nix/nix-${nixVersion}/install \
  | sh -s -- --no-daemon --yes
```

The `nixbld` group, build users, and `/nix` directory are created manually before the installer runs â€” this pre-satisfies the installer's expectations and allows `--yes` (non-interactive) to work as root. `ENV PATH` is placed before `RUN` so the installed `nix` binary is on `PATH` for subsequent Dockerfile layers. Both `/root/.nix-profile/bin` and `/nix/var/nix/profiles/default/bin` are included because the installer may place the `nix` binary in either location depending on profile state.

**Volume pre-population note:** Podman (like Docker) pre-populates an empty named volume from the image's directory contents on the first container start. Installing Nix into `/nix` during the build therefore seeds the volume automatically on first run â€” the `nix` binary and its store entries are available immediately, and the volume grows as `nix develop` downloads further packages.

### `flake.nix` â€” `composeFile` (agent service)

Add to the agent service `environment` block:

```yaml
NO_PROXY: cache.nixos.org,github.com,raw.githubusercontent.com,objects.githubusercontent.com
```

`*.github.com` is enumerated explicitly (rather than using a glob) because not all HTTP clients honour wildcard `NO_PROXY` entries; `raw.githubusercontent.com` and `objects.githubusercontent.com` are the specific GitHub hostnames Nix fetches from.

Add to the agent service `volumes` list:

```yaml
- ${nixVolumeName}:/nix
```

### `flake.nix` â€” `composeFile` (top-level volumes)

Add to the `volumes` section:

```yaml
${nixVolumeName}:
```

### `flake.nix` â€” `composeFile` (agent service â€” LAGUN_AGENT env var)

Add `LAGUN_AGENT: "1"` as the first entry in the agent service `environment` block:

```yaml
environment:
  LAGUN_AGENT: "1"
  HTTP_PROXY: ...
```

### `flake.nix` â€” `shellHook` (LAGUN_AGENT guard)

Wrap the `gitHooks.shellHook` call so it only runs on the host:

```sh
if [ -z "${LAGUN_AGENT:-}" ]; then
  ${gitHooks.shellHook}
fi
```

This prevents `git-hooks.nix` from installing pre-commit hooks into `.git/hooks/` when `nix develop` is invoked inside the agent container.

### `flake.nix` â€” `dockerfile` (CMD)

Change the final `CMD` line from:

```dockerfile
CMD ["claude"]
```

to:

```dockerfile
CMD ["nix", "develop", "--command", "claude"]
```

In the compose workflow this CMD is overridden by `command: sleep infinity`, so it has no runtime effect. It is updated for consistency so that running the image directly (outside compose) also launches Claude Code inside the dev shell.

### `flake.nix` â€” `upStack` CLI (exec hint)

The container continues to use `command: sleep infinity` in the compose file (so it stays alive independently of any single session). The exec hint shown after `run-agent-stack-in-podman` must change from:

```
podman exec -it ${agentContainerName} claude
```

to:

```
podman exec -it ${agentContainerName} nix develop --command claude
```

`nix develop` reads the consuming project's `flake.nix` from the mounted `/workspace`, activates the dev shell (putting all declared packages on PATH and sourcing the shell hook), then launches `claude` inside that environment. This gives the agent exactly the same tooling as a human developer running `nix develop` on the host.
